# Email Provider — Integration Deep-Dive

**Integration**: Strategy (SendGrid, Resend, or AWS SES).  
**Version**: 1  
**Status**: Implementation-grade

---

## 1. Purpose

Specify **transactional email** for the AI Dutch Coach: verification email, password reset, optional subscription receipts, and trial-ending reminder. Covers adapter design, auth, templates, failure handling, and local dev (no provider or mock).

---

## 2. Core Concept

- **Provider**: Backend sends email via provider API (SMTP or REST). We do not receive webhooks for email (delivery/open optional later). Templates can be provider-hosted or our own (HTML/text).
- **Use cases**: Auth (verification, reset); optional billing (receipts, trial ending); optional re-engagement.

---

## 3. Why This Integration Exists

- **Auth**: Email verification and password reset require sending links by email (BFR-009, security).
- **Trust**: Receipts and trial reminders improve trust and conversion (BFR-013).

---

## 4. Business Capabilities Enabled

- **Verification**: Send link to verify email on signup.
- **Password reset**: Send time-limited reset link.
- **Receipts**: Optional send on invoice.paid (Stripe).
- **Trial ending**: Optional reminder 1 day before trial_ends_at.

---

## 5. Scope

### 6. In Scope

- Send transactional email (to, subject, body or template_id, reply_to).
- Templates: verification, reset, receipt, trial_ending (provider templates or our HTML).
- Adapter interface; one implementation per provider (SendGrid, Resend, SES).
- Local dev: mock adapter (log or capture in memory); no send. Optional: provider test mode or sandbox.

### 7. Out of Scope

- Marketing campaigns; inbound email parsing; webhooks for delivery/open (optional later).

---

## 8. Triggering Flows / Usage Points

| Trigger | Flow |
|---------|------|
| User signs up (email) | Auth service → queue or sync → EmailAdapter.send(verification_template, to, { verify_link }) |
| User requests password reset | Auth → generate token → EmailAdapter.send(reset_template, to, { reset_link }) |
| Stripe invoice.paid (optional) | Webhook handler → EmailAdapter.send(receipt_template, user_email, { amount, date }) |
| Trial ending tomorrow (job) | Cron/job → find users with trial_ends_at = tomorrow → EmailAdapter.send(trial_ending_template, to, { trial_ends_at }) |

---

## 9. Inputs

- **Send**: to (email), subject (or template_id), body/template_vars, from (env), reply_to (optional).

---

## 10. Outputs

- **Success**: Provider returns 202/200; we log message_id for debugging.
- **Failure**: Provider 4xx/5xx; we log and retry (queue) or return error to caller.

---

## 11. Data Domains Involved

- **Auth**: verification_tokens, reset_tokens (our DB); email sent log optional.
- **User**: email from profile for receipts and trial reminder.

---

## 12. Source of Truth Rules

- **Links**: Verification and reset links are generated by our backend (signed token or random token stored in DB); email only delivers the link. Provider is not source of truth for auth state.

---

## 13. Authentication Model

- **Provider**: API key (SendGrid, Resend) or IAM (SES). Stored in env (SENDGRID_API_KEY, RESEND_API_KEY, or AWS_*). Backend only.

---

## 14. Authorization / Consent Model

- **Transactional**: Verification and reset are part of auth flow; no separate consent. Receipts and trial reminder: optional; respect user email preferences (e.g. unsub from marketing; transactional may still be sent per policy).

---

## 15. Configuration Model

| Key | Type | Description |
|-----|------|-------------|
| EMAIL_PROVIDER | string | sendgrid \| resend \| ses |
| SENDGRID_API_KEY / RESEND_API_KEY / AWS_* | string | Provider key |
| FROM_EMAIL | string | Sender address |
| SUPPORT_EMAIL | string | Reply-to or support |
| VERIFICATION_URL_BASE | string | Base URL for verify link (e.g. https://app.../verify?token=) |
| RESET_URL_BASE | string | Base URL for reset link |

---

## 16. Environment Strategy

| Env | Setup |
|-----|--------|
| **Local** | Mock adapter: no real send; log or capture in array for tests. EMAIL_MOCK=true. |
| **Staging** | Provider test/sandbox key; real send to test addresses or provider inbox. |
| **Production** | Live key; real send; FROM_EMAIL and domain verified in provider. |

---

## 17. Data Flow Design

- **Sync**: Auth flow may call send synchronously and wait for 202 (fire-and-forget acceptable) or enqueue email job and return immediately (preferred for reset: “If that email exists, we sent a link”).
- **Async**: Preferred for all sends: enqueue job with (template, to, vars); job calls adapter; on failure job retries.

---

## 18. Sync / Polling / Webhook Design

- **Outbound only**: We send; no webhooks required for MVP. Optional: delivery/open webhooks from provider for analytics.

---

## 19. Failure Handling

| Failure | Handling |
|---------|----------|
| Provider 5xx / timeout | Retry (job or sync retry) 2–3 times with backoff. |
| Provider 4xx (invalid, bounce) | Log; do not retry; optional mark user email invalid. |
| Rate limit (429) | Retry after Retry-After; backoff. |

---

## 20. Retry Strategy

- **Job**: 3 attempts, exponential backoff. Dead-letter after max; alert.
- **Sync**: One retry after 2s for 5xx; then return error to caller.

---

## 21. Rate Limiting / Quota Considerations

- **Provider**: Per-key limits (e.g. SendGrid 100/day on free). Apply per-user rate limit for verification/reset (e.g. 3 emails per 15 min per email address) to prevent abuse.

---

## 22. Security / Compliance Requirements

- **Secrets**: API key in env/vault. Links: use short-lived tokens (e.g. 1h reset, 24h verify); HTTPS only.
- **PII**: Email address and link in provider; do not log full link in plaintext (log token id only). GDPR: lawful basis for transactional email (contract/legitimate interest); allow unsub for marketing (if any).

---

## 23. Auditability / Logging Requirements

- Log: Template, to (hash or redact), result (sent/failed), message_id. Do not log full link or token.

---

## 24. Observability / Monitoring

- **Metrics**: Sent count by template; failure count; latency.
- **Alerts**: Failure rate spike; provider 5xx.

---

## 25. UI / UX Implications

- **Verification**: “Check your email for a verification link.” Show same message whether or not email exists (no enumeration).
- **Reset**: “If an account exists, we sent a reset link.” Same.
- **Errors**: “We couldn’t send the email. Try again later.”

---

## 26. Admin / Operations Implications

- **Templates**: Manage in provider dashboard or in repo (HTML files). Version and test before deploy.
- **Bounces**: Optional: handle bounce webhook to mark email invalid and stop sending.

---

## 27. API / Adapter Design

- **Interface**: `EmailAdapter.send(templateId, to, vars) → { messageId? }`. Implementations: SendGridAdapter, ResendAdapter, SESAdapter, MockEmailAdapter (log/capture).
- **Templates**: verification, password_reset, receipt, trial_ending. Vars: link, user name, etc.

---

## 28. Event / Async Flow Design

- **Trigger**: Auth or webhook or cron triggers “send email”; enqueue job; job calls adapter. Return 200 to user immediately for reset/verification (“check your email”).

---

## 29. Data Persistence Requirements

- **Tokens**: verification_tokens, password_reset_tokens in DB (token hash, user_id, expires_at). Optional: email_sent_log (template, to_hash, sent_at) for debugging.

---

## 30. Local Development Setup

- **Mock**: Set EMAIL_MOCK=true; adapter logs or pushes to in-memory array; tests assert “email” would have been sent with correct link. No provider key required.
- **Real**: Set provider key and FROM_EMAIL; send to own test inbox. Use provider sandbox if available (e.g. SendGrid sandbox mode).

---

## 31. Testing Requirements

- **Unit**: Mock adapter; assert send called with correct template and vars (e.g. link contains token).
- **Integration**: With mock or test key: trigger verification flow; assert email “sent” (mock) or receive in test inbox (real). Reset flow: assert link in vars.

---

## 32. Rollout / Feature Flag Strategy

- **Feature flag**: “email_verification_required” to gate verification; “email_receipts_enabled” for receipts. Use mock in CI.

---

## 33. Example Scenarios

**Verification**: User signs up with email → Backend creates verification token → Enqueue send verification email (link=https://app.../verify?token=xxx) → User clicks → Backend validates token → Marks email verified.

**Reset**: User requests reset → Backend creates reset token (1h) → Enqueue send reset email → User clicks link → Backend validates token → Shows reset form → User submits new password.

---

## 34. Edge Cases

- **User changes email before verifying**: Old link invalid; new verification sent to new email if requested.
- **Rate limit**: Per-email rate limit prevents spam; return “Try again in 15 minutes” or same generic message.
- **Provider down**: Job retries; user sees “Try again later” if sync, or no immediate feedback if async (email arrives when provider recovers).

---

## 35. Recommended Technical Design

- **Service**: EmailService with send(template, to, vars); enqueue to job; job calls EmailAdapter. Adapter selected by EMAIL_PROVIDER env.
- **Templates**: Store in provider (SendGrid/Resend) or as files; inject vars server-side. Use HTTPS links only.

---

## 36. Suggested Implementation Phasing

- **Phase 1**: Mock adapter; verification and reset templates; enqueue send; no real provider in dev.
- **Phase 2**: Connect one provider (Resend or SendGrid); staging and prod; receipts and trial_ending optional.
- **Phase 3**: Bounce handling; delivery analytics if needed.

---

## 37. Summary

**Email** is **strategy-based** (SendGrid, Resend, SES). **Local dev** uses **mock adapter** (no key). Verification and password reset are required; receipts and trial reminder optional. Send asynchronously via job; retry on failure. Security: short-lived tokens, no logging of full links; rate limit per email to prevent abuse.
