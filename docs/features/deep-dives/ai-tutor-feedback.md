# AI Tutor Feedback — Deep-Dive Specification

## 1. Purpose

AI Tutor Feedback provides post-activity feedback (grammar, vocabulary, pronunciation, fluency, listening) after lessons, scenario conversations, and voice sessions. Feedback is generated from session/lesson data, displayed to the user, and persisted for history. When feedback is AI-generated, it must be indicated (IS-016) and moderated (IS-017). This spec covers FD-11: triggers, categories, persistence, and integration with FD-02, FD-03, FD-04, FD-05.

## 2. Core Concept

- **Feedback**: Structured summary of strengths, areas to improve, and optional “Practice this” next steps; categories: grammar, vocabulary, pronunciation, fluency, listening (as applicable per activity).
- **Trigger**: End of scenario (FD-03), voice session (FD-04), or lesson (FD-02); backend aggregates turns, errors, pronunciation score, and generates feedback (rule-based or AI).
- **Persistence**: Feedback linked to session/lesson for history and progress; optional “Practice this” links to specific lesson or skill.

## 3. Why This Feature Exists

- **Learning efficacy**: Learners understand what to work on next.
- **Trust**: IS-016 (AI feedback indicated); IS-017 (moderation) ensure transparency and safety.
- **Differentiation**: Premium can get richer feedback (e.g. pronunciation detail).

## 4. User / Business Problems Solved

- Learners see clear, actionable feedback after practice.
- Business reinforces value and supports skill-based recommendations (Personalization).

## 5. Scope

### 6. In Scope

- Generate and display post-activity feedback for grammar, vocabulary, pronunciation, fluency, listening as applicable (FD11-FR-001).
- Indicate when feedback is AI-generated (IS-016) (FD11-FR-002).
- Persist feedback and link to session/lesson for history (FD11-FR-003).
- Apply moderation to AI-generated feedback text (IS-017) (FD11-FR-004).
- Optional “Practice this” CTA linking to lesson or skill practice.
- Trigger from scenario end, voice end, lesson end; optional after listening (FD-05).

### 7. Out of Scope

- In-session corrections (e.g. during conversation); only post-activity summary here.
- Content of recommended “Practice this” lesson (Lesson Engine/Personalization); this spec only stores link or content_id.
- Pronunciation scoring mechanics (FD-06); feedback consumes pronunciation result if available.

## 8. Main User Personas

- **All learners**: See feedback after core activities; free gets basic, premium richer (e.g. pronunciation breakdown).
- **Exam candidate**: Feedback may reference exam components (e.g. “Focus on B1 listening”).

## 9. Main User Journeys

| Journey | Steps |
|--------|--------|
| **After scenario** | End conversation → Summary screen with feedback (grammar, vocabulary, fluency); “AI-generated” label; “Practice grammar” CTA. |
| **After voice** | End voice session → Summary + feedback (pronunciation, fluency, vocabulary); “Practice pronunciation” CTA. |
| **After lesson** | Complete lesson/quiz → Feedback (vocabulary, grammar); “Next lesson” or “Practice weak skill.” |
| **History** | Profile or Activity → List past feedback; tap to see detail. |

## 10. Triggering Events / Inputs

- **Scenario end**: POST /conversation/end returns or triggers feedback; input: session_id, turns, errors (if any).
- **Voice end**: POST /voice/end returns or triggers feedback; input: session_id, turns, optional pronunciation result (FD-06).
- **Lesson complete**: POST /progress/lesson (FD-02) can trigger feedback; input: lesson_id, score, answers.
- **Listening complete**: POST /listening/attempts (FD-05) can trigger feedback; input: exercise_id, score, answers.
- **Generation**: Backend aggregates data; rule-based (e.g. score bands, error counts) or LLM (with moderation); store feedback record; return in response or via GET /feedback/:id.

## 11. States / Lifecycle

- **Pending**: Activity ended; feedback generation started (sync or async).
- **Ready**: Feedback generated and stored; shown to user.
- **Viewed**: User saw feedback (optional tracking for analytics).
- **History**: Persisted; user can revisit from profile/activity history.

## 12. Business Rules

- **IS-016**: When feedback is AI-generated, UI must indicate (e.g. “Generated by AI” or icon).
- **IS-017**: AI-generated feedback text must pass moderation before display and storage.
- **Categories**: grammar, vocabulary, pronunciation, fluency, listening — show only applicable (e.g. no pronunciation for text-only scenario if not analyzed).
- **Premium**: Richer feedback (e.g. pronunciation detail, more granular tips) for premium; free gets basic summary.
- **Link**: Feedback can reference content_id or skill for “Practice this”; resolved by Lesson Engine or Personalization.

## 13. Configuration Model

- **Feedback template**: Per activity type (lesson, scenario, voice, listening); rule-based mapping (score → message) or LLM prompt.
- **Categories**: Fixed list; which categories per activity type (e.g. voice → pronunciation, fluency, vocabulary).
- **Moderation**: Same as FD-03; filter before store and display.
- **Premium vs free**: Which fields or depth of feedback for free vs premium; config.

## 14. Data Model

- **feedback_records**: id, user_id, source_type (lesson|scenario|voice|listening), source_id (lesson_id, session_id, etc.), categories (JSONB: { grammar: { summary, score?, tips[] }, vocabulary: {...}, ... }), is_ai_generated (boolean), created_at. Optional: content_id for “Practice this.”
- **Moderation**: AI-generated text fields moderated before write; store only moderated content.
- **History**: List by user_id, source_type; paginated for profile/activity.

## 15. Read Model / Projection Needs

- **Post-activity**: Single feedback for current activity (returned in end response or GET /feedback/latest?source_id=).
- **History**: List feedback_records for user; filter by source_type; for “Past feedback” or activity log.
- **Personalization**: Optional; weak areas from feedback can feed skill profile (e.g. “grammar weak” from multiple feedbacks).

## 16. APIs / Contracts

| Method | Endpoint | Description | Request | Response |
|--------|----------|-------------|---------|----------|
| GET | `/v1/feedback/:id` | Get feedback by id | — | 200 { feedback }; 404 |
| GET | `/v1/feedback` | List user’s feedback (history) | Query: source_type?, limit, offset | 200 { feedbacks[] } |
| GET | `/v1/feedback/latest` | Get latest feedback for a source (e.g. after end) | Query: source_type, source_id | 200 { feedback }; 404 |
| (Embedded) | In POST /conversation/end, /voice/end | Feedback in end response | — | 200 { summary, feedback_id?, feedback? } |

**Generation**: Can be synchronous (end response includes feedback) or async (return feedback_id; client polls GET /feedback/:id or gets on next load). Moderation applied before storing.

## 17. Events / Async Flows

- **feedback_generated**: feedback_id, source_type, source_id (internal or analytics).
- **feedback_viewed**: feedback_id, user_id (analytics).
- **feedback_practice_clicked**: feedback_id, content_id (analytics; when user taps “Practice this”).
- **Async**: If generation is async (e.g. LLM), queue job; when done, store and optionally push “Your feedback is ready”; client can poll.

## 18. UI / UX Design

- **Feedback card**: Summary per category (e.g. “Grammar: Good”; “Vocabulary: Focus on ordering phrases”); tips list; “Generated by AI” badge when applicable (IS-016).
- **Practice CTA**: “Practice grammar” → link to lesson or skill filter; optional content_id from feedback.
- **History**: List of past feedback with date and source (e.g. “Café scenario, Mar 12”); tap to expand.
- **Empty**: If generation failed or not applicable, show “No feedback this time” or only summary without categories.

## 19. Main Screens / Components

- **FeedbackCard**: Categories, scores/tips, AI badge; used on scenario/voice/lesson summary screens.
- **FeedbackHistoryScreen**: List of feedback_records; filter; expand for detail.
- **PracticeCTA**: Button linking to content_id or skill; used inside FeedbackCard.

## 20. Permissions / Security Rules

- **Authenticated**: All feedback endpoints require auth; user only sees own feedback.
- **Source ownership**: feedback.source_id must belong to user (session/user_id, lesson progress/user_id); validate when generating and when reading.
- **Moderation**: Never store or return unmoderated AI-generated text (IS-017).

## 21. Notifications / Alerts / Side Effects

- **In-app**: Feedback shown immediately after activity end (if sync) or “Your feedback is ready” (if async).
- **Optional push**: “See your feedback from today’s conversation” (Notifications).
- **Personalization**: Feedback weak areas can feed skill profile for recommendations (optional).

## 22. Integrations / Dependencies

- **FD-02 (Core Lessons)**: On lesson complete, trigger feedback generation; input: lesson_id, score, answers.
- **FD-03 (Scenarios)**: On conversation/end, trigger feedback; input: session, turns; persist and return or async.
- **FD-04 (Voice)**: On voice/end, trigger feedback; input: session, optional pronunciation (FD-06).
- **FD-05 (Listening)**: On attempt submit, optional feedback; input: exercise_id, score.
- **FD-06 (Pronunciation)**: Pronunciation result as input for voice feedback category.
- **Moderation**: Same pipeline as FD-03; apply to AI-generated feedback text (IS-017).
- **Lesson Engine / Personalization**: Resolve “Practice this” content_id to lesson or skill URL.

## 23. Edge Cases / Failure Cases

- **Generation fails**: Show summary only (e.g. “You completed the scenario”) without detailed feedback; optional “Feedback unavailable, try again later.”
- **Moderation blocks**: Replace AI feedback with generic safe message or skip AI part; store only safe content.
- **Async delay**: If feedback not ready in end response, return feedback_id and “Feedback generating”; client polls or shows “View feedback” when ready.
- **No data**: e.g. very short voice session; minimal or no feedback; show “Complete a longer session for detailed feedback.”
- **Premium vs free**: Free gets basic (e.g. overall score + one tip); premium gets full categories; 403 not needed if same endpoint, just fewer fields.

## 24. Non-Functional Requirements

- **Latency**: Sync generation < 5s so end response is acceptable; else async and poll.
- **Availability**: Feedback read for history; generation depends on LLM/moderation if AI; degrade with basic rule-based feedback if LLM down.
- **Moderation**: All AI output must pass before display (IS-017).

## 25. Analytics / Auditability Requirements

- **Events**: feedback_viewed, feedback_practice_clicked. Include feedback_id, source_type; no PII in payload.
- **Audit**: Feedback stored for user history and support; retention per Data doc.

## 26. Testing Requirements

- Unit: Category selection per source type; moderation stub (pass/block); premium vs free fields.
- Integration: Trigger from scenario/voice/lesson end; GET feedback by id and list; 404 for other user’s feedback.
- E2E: Complete scenario → see feedback with AI badge; tap “Practice this” → navigate to lesson.

## 27. Recommended Architecture

- **Feedback service or module**: Receives trigger (source_type, source_id, raw data); aggregates; generates (rules or LLM); runs moderation on AI text; persists feedback_records; returns or exposes via GET. Called by Conversation, Voice, Lesson Engine, Listening handlers after they persist completion.
- **Storage**: PostgreSQL feedback_records; optional cache for “latest” by source_id.

## 28. Recommended Technical Design

- **Generation**: Rule-based first (score bands, error counts) for latency and reliability; LLM for richer text when needed; always moderate LLM output.
- **Idempotency**: One feedback per (source_type, source_id); if trigger called again (e.g. retry), upsert or skip.
- **Practice CTA**: content_id or skill_key stored in feedback; client builds URL (e.g. /lessons?skill=grammar) or backend returns deep link.

## 29. Suggested Implementation Phasing

- **Phase 1**: Rule-based feedback for lesson and scenario (scores, 1–2 tips); persist and return in end response; IS-016 badge when applicable (e.g. scenario = AI).
- **Phase 2**: Voice and listening feedback; pronunciation category from FD-06; moderation on AI feedback; GET history.
- **Phase 3**: Richer AI-generated feedback; “Practice this” CTA; premium vs free depth; analytics.

## 30. Summary

AI Tutor Feedback provides post-activity feedback (grammar, vocabulary, pronunciation, fluency, listening) after lessons, scenarios, voice, and listening. It is generated from session/lesson data, indicated when AI-generated (IS-016), and moderated (IS-017). Feedback is persisted and linked to source for history; optional “Practice this” links to lessons. Implementation must integrate with all completion flows, apply moderation to AI text, and respect premium vs free depth.
