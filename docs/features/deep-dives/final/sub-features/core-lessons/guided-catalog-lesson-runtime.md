# Sub-Feature: Guided catalog lesson runtime (strict bundle)

**Feature**: Core Lessons (FD-02) — CEFR curriculum path extension  
**Sub-feature**: guided-catalog-lesson-runtime  
**Parent deep-dives**: `docs/features/deep-dives/core-lessons.md`, `docs/features/deep-dives/cefr-curriculum-path.md`  
**Status**: Draft — aligned to current repo implementation (Next.js client + bundled JSON)

---

## 1. Purpose

Run a **single guided lesson** whose structure, copy, interactions, and skill metadata come from a **versioned curriculum bundle** (not ad-hoc mocks). This sub-feature defines how the client resolves `lessonId`, maps bundle steps to UI, bridges guided practice to a four-skills wrap-up, records weak tags for personalization, and enqueues review items—without silent fallbacks when the catalog is the source of truth.

---

## 2. Core Concept

- **Bundle**: `data/curriculum/nl-NL/A2/catalog.bundle.json` — generated by `scripts/generate_a2_nl_curriculum.py`, typed and imported in `src/demo-data/curriculum/a2Catalog.ts`.
- **Runtime**: Route `src/app/app/learn/[lessonId]/page.tsx` renders `GuidedLessonPage` (`src/features/lessons/GuidedLessonPage.tsx`).
- **Resolution**: `getA2LessonRecordById(lessonId)` returns the lesson record or `undefined` if the id is unknown to the bundle.
- **Steps**: Each `lesson_plan.steps[]` entry becomes a **guided step** with `learner_title`, `activity` (markdown), optional `interaction`, `illustration`, `skill_focus`, `listening_level`, `common_error_tags`, etc.
- **Strictness**: Quiz and flashcards pages also resolve content via `a2Catalog`; there is **no** mock lesson body for A2 when the id exists in the catalog (see implementation for edge cases when list metadata still uses `MOCK_LESSONS`).

---

## 3. User Problems Solved

- Learners follow **consistent, level-scoped** lesson flows authored in the curriculum pipeline.
- Authors change JSON/regenerate bundle; app picks up structure without hand-coded per-lesson screens.
- Listening, grammar, guided practice, self-check, and four-skills blocks render from **one schema** (`A2LessonPlanStep`).

---

## 4. Trigger Conditions

- User navigates to `/app/learn/{lessonId}` (or equivalent) from path panel, catalog, or deep link.
- `lessonId` matches a `lesson.id` present in `catalog.bundle.json` (via `A2_LESSON_RECORDS` / lookup map).

---

## 5. Inputs

- **Route param**: `lessonId` (string).
- **Bundle**: Imported JSON + TypeScript types from `a2Catalog.ts`.
- **Optional list metadata**: `MOCK_LESSONS` (or future API) for title/thumbnail when browsing—**must not** contradict bundle id; unknown ids should not appear in catalog UI when strict mode applies.
- **Client storage**: `localStorage` for review queue (`a2ReviewStore`) — keys scoped per device.

---

## 6. Outputs

- Rendered step UI (progress bar, step title, markdown body, illustrations, listen controls, interactions).
- Side effects: `recordWeakSelfCheckTags` when self-check answers are wrong; `enqueueReviewFromLesson` when completing a lesson with self-check tag filter rules.
- Navigation: “Next lesson” or return to path/catalog via router.
- **Flashcards / Quiz**: separate routes consume the same `lessonId` and catalog record (`FlashcardsPage`, `QuizPage`).

---

## 7. Workflow / Lifecycle

1. Parse `lessonId` from route; load `record = getA2LessonRecordById(lessonId)`.
2. Map `record.lesson_plan.steps` to internal `GuidedStep[]` (ids `step-{n}`, content from `activity`, interaction passthrough).
3. **Step loop**: Show current step; specialized renderers for listen content, self-check quiz, warm-up reveal, guided practice pager, grammar/four-skills layouts, freer practice premium panel.
4. **Guided practice → four skills**: When the lesson contains a four-skills markdown block, `GuidedPracticeFourSkillsBridge` / `GuidedPracticePager` avoid stacking pagers; user completes practice then continues to wrap-up.
5. On lesson completion path: enqueue review items; record weak tags from self-check items that expose `common_error_tags`.
6. User exits or navigates to flashcards/quiz as offered by UI.

---

## 8. Business Rules

- Content visible must match **published** curriculum policy (bundle in repo = stand-in for “published” slice).
- Premium/freer-practice steps (`FreerPracticePremiumPanel`) gate UX copy; entitlement resolution remains with FD-12 when backend exists.
- Self-check and quiz behaviors follow FD-02 FRs for feedback and retry where implemented in `InteractiveSelfCheck` / quiz pages.

## 10. Data Model

**In bundle (per lesson record)** — illustrative shape:

| Field | Role |
|-------|------|
| `id` | Stable lesson id (URL segment) |
| `lesson_plan.steps[]` | Ordered steps with `skill_focus`, `interaction`, `activity` |
| `grammar_tracks`, `vocabulary`, `flashcards`, `quiz` | Downstream pages |
| `tags`, `a2_band`, `cefr_level` | Filtering and path UI |

**Client-only** (today): `a2ReviewStore` entries for spaced/review UX; weak tags recorded locally until `activity-ingestion` API exists.

**Target server** (per architecture docs): `lessons`, `lesson_progress`, `lesson_templates` — this sub-feature’s server counterpart will persist checkpoints and completion instead of local-only review.

---

## 17. Permissions & Security

- Auth required if lessons are user-specific (progress, review queue); public read of static bundle is acceptable only for marketing demos.
- **PII**: Lesson content should not embed user data; reflections (FD-07) stay separate.
- **CSP**: Allow image hosts used in `illustration.src`; audio from allowed origins.

---

## 18. Error Handling

- **Unknown `lessonId`**: Show not-found UI; do not fabricate steps from mocks when strict catalog mode is on.
- **Empty `steps`**: Treat as content error; log and show maintenance message.
- **Invalid `interaction` shape**: Renderer falls back to text-only for that step; generator validation should prevent this in CI.
- **Audio failure**: User-visible retry; degrade to text.

---

## 19. Edge Cases

- User refreshes mid-lesson: step index resets to 0 unless progress API restores checkpoint (future).
- Deep link to last step: not supported until progress API exists.
- Very long markdown tables: `LessonStepContent` table parser must not break layout (see component tests).
- **MOCK_LESSONS** mismatch: If list shows a lesson id not in bundle, runtime must fail clearly—catalog UI should filter to bundle ids only.

---

## 20. Performance Considerations

- Bundle size: code-split or lazy-load non-critical lesson assets; consider JSON chunking per level when multiple locales exist.
- **Images**: Use `next/image` where applicable; explicit width/height from bundle.
- Avoid re-parsing markdown on every keystroke; memoize step bodies by `lessonId` + step index.

---

## 21. Observability

- Log: `lessonId`, `step_count`, render errors (no PII).
- Metrics: time-to-complete-lesson, self-check error rate, audio play failures.
- Build-time: generator `validate()` must pass in CI for main branch.

## 22. Example Scenarios

**A — Listening step**  
Step has `skill_focus: "listening"`, `activity` with `**Listening:**` header; user plays slow audio, advances.

**B — Self-check weak tags**  
Item includes `common_error_tags: ["word_order"]`; wrong answer calls `recordWeakSelfCheckTags` for later filtering.

**C — Four-skills bridge**  
Lesson includes `**Listening:**` … `**Speaking:**` block in one step; user completes guided practice items first, then “Continue to four skills wrap-up.”
